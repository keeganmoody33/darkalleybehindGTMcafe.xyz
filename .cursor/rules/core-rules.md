# Core Rules

- Always read the canonical docs and progress.txt before making changes.
- Never start with implementation if requirements are ambiguous.
- Never code before the relevant plan step is identified.
- Work one approved implementation step at a time.
- Do not invent requirements, features, dependencies, routes, schema fields, or design patterns.
- If a change affects architecture, update docs/ADR.md.
- If a change affects data flow, update docs/DATA_FLOW.md.
- If a change adds or changes components, update docs/COMPONENT_MAP.md.
- After every meaningful change, update progress.txt.
- After every repeated mistake or useful correction, update lessons.md.
- Be explicit about assumptions. Surface them before acting.
